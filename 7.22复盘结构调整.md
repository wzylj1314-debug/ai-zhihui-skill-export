# AI 智绘 Skill 化方案复盘与结构调整

## 1. 复盘结论

当前仓库已经完成了现有资产整理和 Skill 化原型，但暂不作为最终验收版本。

主要原因不是工程路径或安装说明不完整，而是当前 8 个 Skill 的边界仍然偏“从已有资产倒推”，还没有充分证明：

- 哪些业务场景真的需要 Skill。
- 哪些内容只是 reference。
- 哪些能力应该作为 tool/script。
- 哪些材料低频、一次性，不需要 Skill 化。
- 业务 Agent 最核心的价值能力如何体现。

下一版应先回到方案层，从真实业务场景倒推能力形态，再决定 Skill 数量和边界。

## 2. Skill 的重新定义

Skill 不应等同于资料夹，也不应等同于脚本入口。

本项目中，Skill 应定义为：

> 面向明确调用者，在重复业务场景中，封装非显而易见的判断、流程或工具编排，并产出稳定结构化结果的能力单元。

一个能力是否应该做成 Skill，需要同时满足以下条件：

| 判断条件 | 说明 |
|---|---|
| 高频复用 | 会被同一类 agent 或多个 agent 反复调用 |
| 触发边界清晰 | agent 能判断什么时候调用它、什么时候不调用 |
| 输入输出明确 | 输入和输出可以稳定描述，便于验收 |
| 需要判断 | 不是简单查资料，也不是单个确定性脚本 |
| 多步流程或工具编排 | 需要分类、判断、引用资料、必要时调用工具 |
| 业务影响明确 | 判断错误会导致漏商机、误承诺、错转人工、污染知识库等问题 |

由此得到资产形态边界：

- **Skill**：稳定业务判断、多步流程、工具编排。
- **Reference**：产品知识、FAQ、真实问法、销售话术、风险规则、排障规则、Prompt 示例。
- **Tool/script**：OCR、FAQ 工作台、QMD 检索、钉钉发送、会话分析 Web。
- **不做**：低频、一次性、没有明确输入输出、无法复用的材料。

## 3. 当前版本存在的问题

### 3.1 Skill 数量来自资产分类，而不是场景证明

当前 8 个 Skill 来自功能路由、功能说明、FAQ、风险、提示词、排障、销售、工具等资产分类。

这种方式有利于资产整理，但不能证明每一类都应该成为独立 Skill。  
例如“功能说明”更像 reference，“工具启动”更像 script，“风险边界”更像横向 policy。

### 3.2 多个 Skill 在真实客服场景中边界重叠

真实客服接待中，用户通常不会明确说自己需要“功能路由”“FAQ”还是“排障”。

一句客户问题可能同时涉及：

- 意图识别
- 功能推荐
- 操作说明
- 失败排查
- 风险转人工

如果拆成多个独立 Skill，agent 反而会先卡在“该调用哪个 Skill”。

### 3.3 工具被包装成 Skill 的风险

OCR、FAQ 工作台、QMD、钉钉、会话分析 Web 是确定性执行工具。

它们应该被 Skill 调用或编排，但不应该本身成为 Skill。  
Skill 的价值不是“启动工具”，而是判断：

- 什么时候需要 OCR。
- OCR 结果是否值得入库。
- QMD 检索结果是否重复。
- 钉钉消息是否应该发送给业务群、产品群还是客服群。

### 3.4 业务 Agent 的核心能力不足

当前版本偏客服回答和产品说明，对业务 Agent 最重要的能力覆盖不足：

> 从真实客户沟通中识别高价值业务信号，并形成可复用判断。

这些信号包括：

- 商机
- 痛点
- 预算
- 周期
- 决策链
- 产品反馈
- 风险
- 下一步动作

这部分不应只是会话分析工具的输出，而应该成为稳定 Skill。

## 4. 真实场景到能力形态的映射

下面先从真实业务场景出发，不预设 Skill 数量。

| 场景 | 调用者 | 任务 | 当前卡点 | 需要的判断 | 输出 | 能力形态 |
|---|---|---|---|---|---|---|
| 客服接待判断 | 客服 Agent、钉钉客服机器人 | 判断用户意图并给出回复/排障/转人工 | FAQ、路由、排障、风险分散 | 意图、功能、风险、下一步 | 标准回复、功能推荐、排障步骤、转人工判断 | Skill |
| 客户沟通信号识别 | 销售 Agent、业务分析 Agent、管理者 | 从客户沟通中识别商机和业务价值 | 有会话工具，但缺业务判断标准 | 商机等级、痛点、预算、周期、决策链、风险 | 业务信号卡片、跟进建议 | Skill |
| 知识沉淀判断 | 运营 Agent、知识库维护 Agent | 判断截图/OCR/会话是否值得入库 | 有工具，但缺入库价值判断 | 是否高频、是否重复、归属类型、是否敏感 | 入库建议、目标 reference、复核建议 | Skill |
| 产品功能说明 | 客服、销售、创作助手 | 查询某功能怎么用 | 是知识查询，不是流程判断 | 功能匹配时引用即可 | 功能解释、步骤、边界 | Reference |
| Prompt 写法指导 | 创作助手、客服 | 改写提示词、给示例 | 有价值，但当前未证明独立高频 | 可先作为客服/创作流程引用 | Prompt 模板、优化建议 | Reference，暂不独立 Skill |
| 工具执行 | 运营 Agent、维护 Agent | OCR、QMD、钉钉、Web 启动 | 确定性执行 | 参数和路径即可 | 执行结果 | Tool/script |

由这张表推导，当前最应该 Skill 化的是前三个场景。

因此，下一版不是保留 8 个 Skill，而是收敛为 3 个核心 Skill。

## 5. 三个核心 Skill 的边界

## 5.1 `zhihui-customer-intent-resolution`

### 调用者

- 客服 Agent
- 钉钉群客服机器人
- 一线运营客服助手

### 解决任务

识别用户意图，并决定应该：

- 直接回答 FAQ。
- 推荐功能。
- 给操作步骤。
- 给排障建议。
- 转人工。

### 为什么必须是 Skill

该场景输入高度不标准，用户往往只说口语化问题，例如：

- “我想把这张衣服换个风格，用哪个？”
- “图又糊了，是不是坏了？”
- “这个能不能商用，价格怎么算？”
- “为什么试衣看起来不像真人？”

如果没有 Skill，agent 会在 FAQ、功能路由、排障、风险规则之间来回跳转，容易漏判风险或答非所问。

### 触发条件

- 用户提出产品使用问题。
- 用户询问功能选择。
- 用户反馈生成失败或效果异常。
- 用户提出价格、版权、合同、账号、API、投诉等敏感问题。

### 非触发条件

- 输入是一整段销售沟通，需要判断商机价值。
- 输入是 OCR/会话摘要，需要决定是否入库。
- 用户只要求执行某个确定性工具。

### 输入

```json
{
  "user_message": "用户原话",
  "conversation_context": "可选，多轮上下文",
  "known_feature": "可选，用户提到的功能",
  "failure_description": "可选，失败现象",
  "attachments_summary": "可选，截图或图片描述"
}
```

### 输出

```json
{
  "intent_type": "功能推荐/操作咨询/FAQ/效果排障/风险转人工/投诉升级",
  "recommended_feature": "可选，功能编号和名称",
  "answer": "面向用户的标准回复",
  "troubleshooting_steps": ["可选，排障步骤"],
  "handoff_required": true,
  "handoff_reason": "触发转人工的原因",
  "confidence": "high/medium/low",
  "evidence": ["引用用户原话或判断依据"]
}
```

### 引用资产

- F01-F27/T01 功能说明
- FAQ
- 真实用户问法
- 排障规则
- 风险边界

### 可调用工具

- QMD 检索：知识不确定时调用。
- 钉钉发送：需要人工升级时调用。

## 5.2 `zhihui-business-signal-extraction`

### 调用者

- 销售 Agent
- 业务分析 Agent
- 会话分析 Agent
- 管理者

### 解决任务

从真实客户沟通中识别业务价值，而不是只总结聊天内容。

### 为什么必须是 Skill

业务信号往往隐藏在自然沟通里。  
客户不会直接说“我是高价值商机”，而是会表达：

- “我们设计师每周要出很多款。”
- “老板想看看能不能接到我们现有流程里。”
- “如果效果稳定，可能整个团队都用。”
- “价格怎么算，能不能先试？”
- “这个功能如果能批量处理就好了。”

这些都不是普通 FAQ，而是商机、预算、采购周期、决策链、产品机会的信号。

如果没有 Skill，这些内容容易被当作普通聊天摘要，无法形成销售和产品可行动判断。

### 触发条件

- 输入是一段销售沟通、客户聊天、电话转写、会议纪要。
- 用户要求分析客户价值、跟进优先级、产品反馈。
- 用户要求生成业务/产品同步摘要。

### 非触发条件

- 用户只是问一个具体产品问题。
- 用户只是要求启动会话分析 Web。
- 输入是 FAQ 草稿，需要判断是否入库。

### 业务信号判断体系

| 信号类型 | 判断标准 | 示例线索 | 输出字段 |
|---|---|---|---|
| 商机等级 | 是否有明确需求、试用意愿、购买动作、团队规模 | “想试一下”“团队都要用”“价格怎么算” | `opportunity_level` |
| 痛点类型 | 客户当前业务阻塞点 | 设计效率、款式开发、打样成本、上新速度、素材生产 | `pain_points` |
| 预算信号 | 是否提到价格、采购、套餐、审批 | “多少钱”“老板批预算”“先试用” | `budget_signal` |
| 周期信号 | 是否有明确时间窗口 | “这周要看”“下个月上新”“马上要提案” | `timeline_signal` |
| 决策链 | 是否出现老板、负责人、设计师、采购、IT | “老板想看”“设计主管决定” | `decision_chain` |
| 产品反馈 | 是否指向功能缺口或体验问题 | “能不能批量”“这个效果不稳定” | `product_feedback` |
| 风险信号 | 是否涉及退款、版权、合同、隐私、投诉 | “能不能保证”“版权归谁” | `risk_flags` |
| 下一步动作 | 是否需要销售、产品、技术、客服介入 | 约演示、发资料、拉群、转产品 | `next_actions` |

### 输入

```json
{
  "conversation": "客户沟通全文",
  "customer_profile": "可选，客户行业/角色/规模",
  "source": "聊天/电话转写/会议纪要",
  "analysis_goal": "商机识别/产品反馈/销售跟进/管理汇总"
}
```

### 输出

```json
{
  "opportunity_level": "A/B/C/None",
  "opportunity_reason": "商机等级依据",
  "pain_points": [
    {
      "type": "设计效率/上新速度/打样成本/素材生产/其他",
      "evidence": "客户原话"
    }
  ],
  "budget_signal": {
    "status": "明确/隐含/无",
    "evidence": "客户原话"
  },
  "timeline_signal": {
    "status": "紧急/近期/长期/不明确",
    "evidence": "客户原话"
  },
  "decision_chain": [
    {
      "role": "老板/设计主管/采购/IT/使用者",
      "evidence": "客户原话"
    }
  ],
  "product_feedback": [
    {
      "feature": "相关功能",
      "feedback_type": "功能缺口/体验问题/效果问题/批量能力/集成需求",
      "evidence": "客户原话"
    }
  ],
  "risk_flags": [
    {
      "type": "价格/合同/版权/隐私/投诉/承诺风险",
      "evidence": "客户原话"
    }
  ],
  "next_actions": [
    {
      "owner": "销售/产品/技术/客服",
      "action": "建议动作",
      "priority": "high/medium/low"
    }
  ],
  "summary_for_sales": "给销售看的摘要",
  "summary_for_product": "给产品看的摘要"
}
```

### 引用资产

- 销售话术
- 客户场景
- 产品功能清单
- 套餐边界
- 风险边界
- 产品反馈分类

### 可调用工具

- 会话分析 Web：处理长文本和生成初版摘要。
- 钉钉发送：将业务摘要或产品反馈同步到对应群。
- QMD 检索：核对已有产品反馈或知识库内容。

## 5.3 `zhihui-knowledge-capture-decision`

### 调用者

- 知识库维护 Agent
- 运营 Agent
- 客服主管

### 解决任务

判断客服截图、OCR 文本、会话摘要、FAQ 草稿是否值得沉淀为可复用知识。

### 为什么必须是 Skill

知识沉淀不是“看到内容就入库”。  
它需要判断：

- 是否高频。
- 是否代表真实用户问法。
- 是否已经存在相似内容。
- 应归入 FAQ、真实问法、排障、风险边界、销售话术，还是不入库。
- 是否涉及敏感信息或错误承诺。

如果没有 Skill，FAQ 工作台只会变成“生成草稿工具”，无法保证知识库质量。

### 触发条件

- 输入来自客服截图/OCR。
- 输入来自会话分析摘要。
- 用户要求判断是否入库。
- 用户要求整理 FAQ 或真实问法。

### 非触发条件

- 用户只是实时问产品问题。
- 用户要求分析商机价值。
- 用户只是启动工具。

### 输入

```json
{
  "source_content": "截图 OCR、会话摘要或 FAQ 草稿",
  "source_type": "截图/OCR/会话摘要/人工草稿",
  "existing_search_results": "可选，QMD 检索结果",
  "operator_notes": "可选，人工备注"
}
```

### 输出

```json
{
  "capture_decision": "入库/暂存/不入库/转人工复核",
  "knowledge_type": "FAQ/真实问法/排障规则/风险边界/销售话术/产品反馈",
  "target_reference": "建议归属文件或 reference",
  "dedupe_result": {
    "is_duplicate": false,
    "similar_items": []
  },
  "sensitivity_check": {
    "has_sensitive_content": false,
    "risk_types": []
  },
  "quality_check": {
    "is_reusable": true,
    "reason": "判断依据"
  },
  "draft": "可提交草稿",
  "review_required": true,
  "review_reason": "需要人工复核的原因"
}
```

### 引用资产

- FAQ
- 真实问法
- 功能说明
- 排障规则
- 风险边界
- 入库规范

### 可调用工具

- OCR/FAQ 工作台
- QMD 检索
- 钉钉通知
- 会话分析 Web

## 6. 为什么是 3 个 Skill，而不是 8 个或 4 个

这次不是预设数量，而是从业务场景推导。

### 为什么不是 8 个

原 8 个 Skill 多数来自资产分类，而非独立业务判断。

| 原 Skill | 问题 | 下一版处理 |
|---|---|---|
| `zhihui-product-router` | 更像客服意图判断中的一个步骤 | 并入 `customer-intent-resolution` |
| `zhihui-feature-playbooks` | 产品知识材料 | 降级为 reference |
| `zhihui-faq-answering` | 客服回答的一部分 | 并入 `customer-intent-resolution` |
| `zhihui-risk-handoff` | 横向安全规则，不是独立业务任务 | 降级为 policy/reference，被 3 个 Skill 引用 |
| `zhihui-prompt-coach` | 暂未证明是独立高频业务 Skill | 降级为 reference，后续按调用量评估 |
| `zhihui-troubleshooting` | 客服接待中的效果排障分支 | 并入 `customer-intent-resolution` |
| `zhihui-sales-assistant` | 销售话术是 reference，业务判断应归入信号识别 | 并入 `business-signal-extraction` |
| `zhihui-ops-tools` | 工具说明和知识维护混合 | 重组为 `knowledge-capture-decision` |

### 为什么不是 2 个

如果只保留“客服”和“业务分析”两个 Skill，会把知识沉淀判断混入客服或业务分析中。

但知识沉淀有独立的输入、输出和风险：

- 输入来自截图/OCR/草稿/摘要。
- 输出是是否入库、入哪里、是否重复、是否敏感。
- 错误会污染知识库。

因此它应作为独立 Skill。

### 为什么不是 4 个

Prompt 指导和创作优化目前有价值，但还没有证明它在现阶段应独立为 Skill。

它更适合作为：

- 功能说明 reference
- Prompt 示例 reference
- 客服接待中的创作咨询分支

后续如果真实数据证明创作指导是高频独立场景，再拆出 `zhihui-creative-guidance`。

## 7. Reference 与 Tool 的重组建议

## 7.1 Reference

下一版 reference 应按“被哪些 Skill 引用”组织，而不是按当前 Skill 名称组织。

| Reference 类型 | 内容 | 被谁引用 |
|---|---|---|
| 产品功能说明 | F01-F27/T01 | 客服接待、业务信号、知识沉淀 |
| FAQ | 高频问答 | 客服接待、知识沉淀 |
| 真实用户问法 | 口语化问题样例 | 客服接待、知识沉淀 |
| 排障规则 | 失败现象、原因、重试建议 | 客服接待、知识沉淀 |
| 风险边界 | 价格、合同、版权、退款、API、隐私、投诉 | 三个 Skill 都引用 |
| 销售话术 | 价值表达、客户场景、套餐边界 | 业务信号 |
| Prompt 示例 | AI 改款、保留项、修改项、风格描述 | 客服接待，后续可支持创作 Skill |

## 7.2 Tool/script

| Tool/script | 定位 | 被谁调用 |
|---|---|---|
| OCR/FAQ 工作台 | 截图转草稿、审核看板 | 知识沉淀 |
| QMD 检索 | 查询已有知识、去重 | 客服接待、知识沉淀、业务信号 |
| 钉钉发送 | 消息同步、转人工、业务/产品摘要推送 | 三个 Skill 都可能调用 |
| 会话分析 Web | 长会话摘要和分析辅助 | 业务信号、知识沉淀 |

## 8. 明确不做 Skill 的内容

以下内容下一版不做独立 Skill：

| 内容 | 原因 |
|---|---|
| 单个产品功能说明 | 只是知识材料 |
| 单条 FAQ | 只是回答依据 |
| 单个脚本启动说明 | 是工具使用说明 |
| 一次性复盘报告 | 低频材料 |
| 单个 Prompt 示例 | 是 reference |
| 单个风险条款 | 是 policy/reference |
| 未验证高频的创作指导 | 暂不拆 Skill，保留为 reference |

这部分“不做”很重要，目的是避免继续把所有资产都包装成 Skill。

## 9. 下一版交付内容

下一版先交方案层与边界，不继续围绕旧 8 个 Skill 补路径和安装说明。

交付内容：

1. 真实场景—能力形态映射表。
2. 3 个核心 Skill 的触发条件、非触发条件、输入 schema、输出 schema。
3. `business-signal-extraction` 的业务信号分类体系。
4. `knowledge-capture-decision` 的入库价值判断标准。
5. Reference 重组方案。
6. Tool/script 边界说明。
7. 原 8 个 Skill 的合并、降级、废弃清单。
8. 验收样例集设计。

## 10. 验收标准

下一版不以 Skill 数量验收，而以真实业务场景表现验收。

### 10.1 客服接待判断验收

样例建议：不少于 20 条真实客服问题。

覆盖类型：

- 功能推荐
- 操作咨询
- FAQ
- 效果失败
- 价格/合同/版权/API/投诉等风险转人工

通过标准：

- 能给出意图分类。
- 能给出用户可直接看到的回复。
- 涉及风险时必须转人工。
- 不能擅自承诺价格、合同、版权、退款、私有化等内容。

### 10.2 业务信号识别验收

样例建议：不少于 10 段真实销售或客户沟通。

覆盖类型：

- 明确商机
- 弱商机
- 无商机
- 产品反馈
- 风险投诉
- 决策链信息

通过标准：

- 每段沟通输出商机等级。
- 每个重要判断必须引用客户原话作为证据。
- 能区分销售跟进、产品跟进、客服处理、暂不处理。
- 能输出下一步动作和负责人。

### 10.3 知识沉淀判断验收

样例建议：不少于 20 条截图/OCR/FAQ 草稿。

覆盖类型：

- 可入库 FAQ
- 真实问法
- 排障规则
- 风险边界
- 重复内容
- 敏感内容
- 低价值噪声

通过标准：

- 能判断是否入库。
- 能说明入库类型和目标 reference。
- 能发现重复或相似内容。
- 能标记敏感和人工复核项。
- 不能把低频噪声直接写入知识库。

## 11. 最终结论

当前 8 个 Skill 是资产整理原型，不作为最终验收形态。

下一版应从真实业务场景重新设计，将能力收敛为 3 个核心 Skill：

```text
zhihui-customer-intent-resolution
zhihui-business-signal-extraction
zhihui-knowledge-capture-decision
```

其他资产归位：

```text
产品功能、FAQ、真实问法、销售话术、风险边界、Prompt 示例、排障规则 -> reference
OCR、QMD、钉钉、FAQ 工作台、会话分析 Web -> tool/script
低频、一次性、没有明确输入输出的材料 -> 不做
```

这版调整的核心不是继续证明“现有资产都能做成 Skill”，而是证明：

> 哪些真实业务判断没有 Skill 就跑不稳。

下一版重点应补强业务 Agent 的核心能力：

> 从真实客户沟通中识别高价值业务信号，并形成可复用、可验证、可行动的判断结果。
